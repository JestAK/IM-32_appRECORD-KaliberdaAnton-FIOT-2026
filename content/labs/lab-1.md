## 2. Аналіз предметної області

**2.1 Актуальність теми**

Українська фандаб-спільнота – це середовище людей, що самостійно перекладають та озвучують аніме, серіали й фільми українською мовою. Попит на україномовний контент зростає, а разом з ним і зростає кількість людей.
Разом із тим, у спільноті відсутня централізована платформа для пошуку людей. Актори озвучення, перекладачі та звукорежисери шукають одне одного через Telegram-чати, Discord-сервери та соціальні мережі. Це призводить до втрати часу та неможливості оцінити досвід кандидата до початку співпраці.
Розроблення веб-платформи FanDub Space вирішує цю проблему шляхом надання єдиного простору, де кожен учасник може розмістити своє портфоліо: ім'я, роль, перелік завершених проєктів, контактні дані та додаткову інформацію.

**2.2 Завдання роботи**

- Проаналізувати проблеми організації фандаб-спільноти та визначити потреби її учасників.
- Сформулювати функціональні та нефункціональні вимоги до системи.
- Побудувати Use-case діаграму для відображення сценаріїв взаємодії користувачів із системою.
- Побудувати ER-діаграму для опису структури даних системи.
- Розробити адаптивний веб-застосунок з використанням Next.js, Tailwind та shadcn/ui.
- Реалізувати фіксований хедер з бургер-меню для мобільних пристроїв.
- Реалізувати динамічну фільтрацію учасників за роллю.
- Забезпечити коректне відображення інтерфейсу на пристроях різних типів.

**2.3 Об'єкт роботи**

Об'єктом роботи є процес пошуку учасників фандаб-проєктів та представлення їхнього творчого доробку в цифровому середовищі.

**2.4 Предмет роботи**

Предметом роботи є адаптивний веб-застосунок для створення та перегляду портфоліо учасників фандаб-спільноти.

## 3. Моделювання системи

**3.1 Бізнес-логіка системи**

- Кожен учасник зареєстрований у одній із трьох ролей: актор озвучення (actor), перекладач (translator) або звукорежисер (sound). Роль є незмінним атрибутом профілю.
- Портфоліо учасника обов'язково містить ім'я/псевдонім, роль, короткий опис та контактні дані. Перелік робіт є необов'язковим, але рекомендованим.
- Кожна робота прив'язана до конкретного учасника. Один учасник може мати необмежену кількість робіт у портфоліо.
- Система відображає учасників у вигляді карток. За замовчуванням відображаються усі учасники; при активації фільтра — лише учасники відповідної ролі.
- Контактні дані відображаються безпосередньо в картці без необхідності реєстрації відвідувача.

**3.2 Функціональні вимоги**

- FR-1: Система повинна відображати список учасників у вигляді карток із базовою інформацією (ім'я, роль, роботи, контакт).
- FR-2: Система повинна забезпечувати фільтрацію учасників за роллю: «Усі», «Актори», «Перекладачі», «Звук».
- FR-3: При активації фільтра система повинна відображати лише учасників відповідної ролі без перезавантаження сторінки.
- FR-4: Система повинна відображати контактні дані кожного учасника у вигляді посилання.
- FR-5: Хедер повинен бути зафіксований у верхній частині сторінки при прокрутці.
- FR-6: На мобільних пристроях навігація повинна згортатися у бургер-меню.
- FR-7: Сторінка повинна містити секції: Hero, Учасники, Про проєкт, Долучитись, Footer.

**3.3 Нефункціональні вимоги**

- NFR-1: веб-застосунок повинен коректно відображатись на екранах від 320px до 1920px. Використовуються breakpoints Tailwind: `sm` (640px), `md` (768px), `lg` (1024px).
- NFR-2: інтерфейс не потребує реєстрації для перегляду портфоліо учасників.
- NFR-3: застосунок підтримується у Chrome, Firefox, Safari та Edge останніх версій як на ПК так і на мобільних пристроях.
- NFR-4: хедер використовує атрибут `aria-expanded` для бургер-кнопки; семантична верстка через `<header>`, `<main>`, `<section>`, `<footer>`.
- NFR-5: код розділений на незалежні компоненти (`Header`, `Participants`), дані винесені в окремий масив `PEOPLE` як імпровізована база даних.

**3.4 Use-case діаграма**

![Use-case діаграма](/assets/labs/lab-1/usediagram.png)

Актор `Відвідувач` може виконувати такі сценарії:

- UC-1. Переглянути список учасників: відображаються лише профілі, схвалені модератором.
- UC-2. Фільтрувати за роллю: Усі / Актори / Перекладачі / Звук.
- UC-3. Переглянути профіль учасника.
- UC-4. Переглянути роботи учасника.
- UC-5. Отримати контактні дані учасника.
- UC-6. Зареєструватись: перехід до ролі Учасника.

Актор `Учасник` (розширює Відвідувача) виконує додаткові сценарії:

- UC-7. Увійти в акаунт.
- UC-8. Підтвердити email: включається автоматично при реєстрації.
- UC-9. Створити портфоліо: після збереження автоматично потрапляє в чергу модерації.
- UC-10. Редагувати портфоліо: після змін знову потрапляє в чергу модерації.
- UC-11. Додати роботу до портфоліо.

Актор `Модератор` виконує сценарії модерації:

- UC-12. Переглянути чергу портфоліо, що очікують розгляду.
- UC-13. Схвалити портфоліо: профіль стає видимим у списку учасників.
- UC-14. Відхилити портфоліо з поясненням причини: учасник отримує сповіщення.
- UC-15. Заблокувати учасника: профіль прихований, акаунт деактивовано.

**3.5 ER діаграма**

![ER діаграма](/assets/labs/lab-1/erdiagram.png)

**Сутність Member (Учасник):**

| Поле         | Тип        | Опис                               |
| ------------ | ---------- | ---------------------------------- |
| `id`         | INTEGER PK | Унікальний ідентифікатор           |
| `name`       | VARCHAR    | Ім'я або псевдонім                 |
| `role`       | ENUM       | `actor` \| `translator` \| `sound` |
| `extra_info` | TEXT       | Додаткова інформація про учасника  |
| `contact`    | VARCHAR    | Контактна інформація               |

**Сутність Work (Робота):**

| Поле        | Тип        | Опис                     |
| ----------- | ---------- | ------------------------ |
| `id`        | INTEGER PK | Унікальний ідентифікатор |
| `title`     | VARCHAR    | Назва проєкту            |
| `href`      | VARCHAR    | Посилання на роботу      |
| `member_id` | INTEGER FK | Зовнішній ключ → Member  |

**Сутність ModerationRequest (Запит на модерацію):**

| Поле           | Тип        | Опис                                  |
| -------------- | ---------- | ------------------------------------- |
| `id`           | INTEGER PK | Унікальний ідентифікатор              |
| `member_id`    | INTEGER FK | Зовнішній ключ → Member               |
| `moderator_id` | INTEGER FK | Зовнішній ключ → Moderator            |
| `status`       | ENUM       | `pending` \| `approved` \| `rejected` |
| `reason`       | TEXT       | Причина відхилення (необов'язково)    |
| `created_at`   | TIMESTAMP  | Час створення запиту                  |

**Сутність Moderator (Модератор):**

| Поле    | Тип        | Опис                     |
| ------- | ---------- | ------------------------ |
| `id`    | INTEGER PK | Унікальний ідентифікатор |
| `name`  | VARCHAR    | Ім'я модератора          |
| `email` | VARCHAR    | Email модератора         |

**Зв'язки:**

- Один `Member` має багато `Work` — **1:N**
- Один `Member` має багато `ModerationRequest` — **1:N**
- Один `Moderator` розглядає багато `ModerationRequest` — **1:N**
